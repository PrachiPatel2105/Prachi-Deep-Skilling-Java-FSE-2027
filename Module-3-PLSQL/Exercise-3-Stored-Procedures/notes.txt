STORED PROCEDURE BASICS:
- Named block of PL/SQL code
- Stored in database
- Can be called multiple times
- Pre-compiled and optimized
- More secure than raw SQL
- Can have parameters
- Can raise errors

SYNTAX:
CREATE OR REPLACE PROCEDURE name (params) IS
    v_variable datatype;
BEGIN
    -- code
    COMMIT;
EXCEPTION
    WHEN OTHERS THEN
        ROLLBACK;
END name;
/

PARAMETER TYPES:
IN      - Input (read-only)
OUT     - Output (write-only)
IN OUT  - Both input and output

SCENARIO 1 - PROCESS MONTHLY INTEREST:
Purpose: Calculate and apply interest to savings accounts
Logic:
- FOR each savings account:
  - Calculate interest (balance × rate)
  - Update balance
  - Record transaction
  - COMMIT

Example:
Account Balance: $50,000
Interest Rate: 1.5%
Interest Amount: $750
New Balance: $50,750

Key Concepts:
- FOR LOOP with cursor
- Calculations
- INSERT into Transactions
- DBMS_OUTPUT for display

SCENARIO 2 - EMPLOYEE BONUS:
Purpose: Update salary with bonus percentage
Parameters:
- p_department_id - Which department
- p_bonus_percentage - Bonus % to apply

Logic:
- Validate parameters
- Get department name
- FOR each employee in department:
  - Calculate bonus
  - Update salary
  - COMMIT

Example:
Department: IT (ID: 1)
Bonus: 10%
Employee: John Smith
  Old Salary: $75,000
  Bonus: $7,500
  New Salary: $82,500

Key Concepts:
- Input parameters

SCENARIO 3 - TRANSFER FUNDS:
Purpose: Move money between accounts with validation
Parameters:
- p_from_account_id - Source account
- p_to_account_id - Destination account
- p_amount - Amount to transfer

Validations:
1. Accounts exist
2. Source has sufficient balance
3. Amount > 0
4. Not same account
5. Parameters not null

Logic:
- Validate all conditions
- Check balance
- Deduct from source
- Add to destination
- Record transaction
- COMMIT or ROLLBACK

Error Handling:
- Insufficient balance
- Account not found
- Invalid parameters
- Two-phase update (deduct & add)

Key Concepts:
- Multiple validations
- RAISE_APPLICATION_ERROR with custom messages
- Transaction rollback
- Error handling
- Atomic operations (both succeed or both fail)

ADVANTAGES OF STORED PROCEDURES:
1. Reusability
   - Write once, use everywhere
   - Consistent logic across applications

2. Performance
   - Pre-compiled on server
   - Faster than individual SQL statements
   - Network traffic reduced

3. Security
   - Users call procedure, not raw SQL
   - Can grant procedure access without table access
   - Hide implementation details

4. Consistency
   - Same logic everywhere
   - Changes automatically propagate
   - Prevents custom implementations

5. Maintainability
   - Update in one place
   - All applications benefit
   - Easier to test

ERROR HANDLING:
EXCEPTION
    WHEN NO_DATA_FOUND THEN
        -- Handle when no rows found
    WHEN TOO_MANY_ROWS THEN
        -- Handle when too many rows
    WHEN OTHERS THEN
        -- Catch all errors
        DBMS_OUTPUT.PUT_LINE('Error: ' || SQLERRM);
END;

RAISE_APPLICATION_ERROR:
RAISE_APPLICATION_ERROR(-20001, 'Insufficient balance');

- Negative error number: -20001 to -20999
- Custom error message
- User sees error in exception block

TRANSACTIONS:
COMMIT   - Save all changes
ROLLBACK - Undo all changes

Best practice: COMMIT after all updates successful
